# CaseAxis Database Design

**Status:** Reviewed — approved for Phase 2 migration implementation
**Last updated:** 2026-05-20
**Phase:** 1 complete — design review incorporated

---

## Design Review Summary

A critical design review was conducted after initial schema design. The following changes
were made. Decisions that survived challenge are noted as defended.

### Changes from Review

| Area | Original | Changed To | Reason |
|---|---|---|---|
| Primary key generation | `gen_random_uuid()` (UUID v4) | UUID v7 via application (`uuid-creator`) | Eliminates B-tree fragmentation; DB column type unchanged |
| Case ownership | Both `client_id` and `organization_id` nullable | CHECK constraint: at least one required | Schema must define what a valid case is |
| `case_notes` mutability | Editable with `updated_at` | Immutable; `supersedes_note_id` for corrections | Compliance-oriented systems treat notes as evidence |
| `user_roles` revocation | Row deleted on revoke | `removed_at` + `removed_by` added | Role revocation history is an audit requirement |
| `case_assignments` uniqueness | Application-enforced only | Unique partial index on active assignments | DB must enforce the at-most-one-active-assignment invariant |
| `audit_logs` entity naming | Java class names (`Case`, `CaseTask`) | Domain constants (`case`, `case_task`) | Java class renames must not break audit history |
| `audit_logs` partition readiness | `PRIMARY KEY (id)` | `PRIMARY KEY (id, occurred_at)` | Range partitioning by `occurred_at` requires the partition key in the PK |
| Case number format | `CA-YYYY-NNNNNN` (global sequence, year cosmetic) | `CA-NNNNNN` | Year in the number implies year-scoped sequence; global sequence with no year reset is cleaner |

### Defended Decisions (survived challenge)

- Lookup tables over PostgreSQL ENUMs
- `assigned_to_id` denormalization on `cases`
- JSONB for `audit_logs.old_values` / `new_values`
- `case_status_history` as a first-class table separate from audit logs
- Soft delete strategy for all business tables
- `TIMESTAMPTZ` everywhere
- No SLA, queue, or tag tables in Phase 1

## Overview

This document defines the initial relational schema for CaseAxis. It covers all core
business domains required for Phase 2 migration implementation: identity and access,
organizations and clients, cases, assignments, status history, tasks, notes, attachment
metadata, and audit logging.

**Design targets:**
- PostgreSQL on a single instance (modular monolith)
- 1,000+ users, 50,000+ organizations, 500,000+ cases, millions of related records
- Normalized relational schema with strong referential integrity
- Auditability as a first-class requirement

---

## Design Principles

### Normalize first, denormalize with justification

The default is third normal form. Controlled denormalization (e.g., caching the current
assignee on the `cases` table) is permitted when the query benefit is clear and the
redundancy is documented.

### Referential integrity is mandatory

Every foreign key relationship is enforced at the database level. Application layer
validation alone is insufficient.

### Lookup tables over enums

Status values, priority codes, and task states are stored in lookup tables, not
PostgreSQL `ENUM` types. Lookup tables allow new values to be added via a seed migration
without altering the schema. They support metadata that enums cannot carry (display
labels, sort order, `is_terminal` flags, `is_initial` flags). ENUMs require
`ALTER TYPE ... ADD VALUE` which takes a brief ACCESS EXCLUSIVE lock in production.

### History tables are immutable

`case_status_history` and `case_assignments` are append-only. Records are never updated
or soft-deleted. They are the operational ledger.

### Audit logs are immutable and unconstrained by FK

`audit_logs` has no foreign key on `entity_id` because it must retain records for
deleted entities. It is never soft-deleted, hard-deleted, or updated.

### Notes are evidence, not scratchpads

`case_notes` are immutable once created. In compliance-oriented case management, a
note represents an observation at a specific point in time. Edits obscure the timeline.
Corrections use the `supersedes_note_id` pattern.

## Primary Key Strategy

**All tables use UUID (128-bit) as the primary key type.**

**Generation strategy: UUID v7** (time-ordered epoch-based UUID) generated by the
application layer using the `uuid-creator` Java library
(`com.github.f4b6a3:uuid-creator`).

```java
// In Spring service layer, not DB default
UuidCreator.getTimeOrderedEpoch()  // UUID v7
```

The PostgreSQL column type remains `UUID`. No database-level default is set for the PK
— the application always supplies the value before insert.

**Why UUID v7 over v4:**

UUID v4 is randomly distributed. On high-insert tables (`audit_logs`, `cases`,
`case_status_history`), random UUID inserts cause constant B-tree page splits in the PK
index. At millions of audit records this produces measurable write amplification and
index bloat. PostgreSQL autovacuum cleans dead tuples but cannot undo live-write
fragmentation.

UUID v7 is time-ordered (monotonically increasing within a millisecond). Inserts go to
the end of the B-tree, matching the behavior of a BIGSERIAL sequence. This eliminates
fragmentation without sacrificing UUID's collision safety or API exposure protection.

**Why UUID over BIGINT:**

CaseAxis is an internal enterprise application behind authentication. Sequential ID
enumeration is a lower risk than in public APIs. However, UUIDs remain the correct
choice because:
- IDs appear in URLs, log files, and API responses
- UUID consistency across all tables simplifies the codebase
- UUID v7 eliminates the performance argument for BIGINT

**Why not BIGINT + separate public UUID:**

The "dual ID" pattern (internal BIGINT PK + external UUID for API) adds a column to
every table, complicates JPA entity mapping, and adds cognitive overhead. The benefit
does not justify the complexity in a modular monolith.

## Timestamp and Metadata Standards

### Timestamp type

All timestamps use `TIMESTAMPTZ` (timestamp with time zone). UTC is stored. Never use
`TIMESTAMP WITHOUT TIME ZONE`.

### Standard columns on all mutable business tables

```
created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
created_by   UUID        NOT NULL REFERENCES users(id)
updated_by   UUID                 REFERENCES users(id)
```

`updated_at` is maintained by a shared PostgreSQL trigger function applied to each
mutable table. The application must not be relied upon to set it.

```sql
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

Applied per table:

```sql
CREATE TRIGGER trg_set_updated_at
BEFORE UPDATE ON <table_name>
FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

**Exception — `created_by` nullable bootstrap case:** The first ADMIN user has no
preceding user to reference. `created_by` on `users` is nullable to accommodate this.
All other inserts must supply `created_by`.

### Immutable tables

Append-only tables carry only `created_at` (or the equivalent event timestamp). No
`updated_at`, no soft-delete columns. This applies to: `case_status_history`,
`case_assignments`, `case_notes`, `audit_logs`.

## Soft Delete Strategy

### Applied to business entities

| Table | Approach | Reason |
|---|---|---|
| `users` | Soft delete | Legal/HR retention; audit references must not orphan |
| `organizations` | Soft delete | Cases reference organizations; hard delete breaks FK integrity |
| `clients` | Soft delete | Cases reference clients; same reason |
| `cases` | Soft delete | Core business record; regulatory retention in real systems |
| `case_tasks` | Soft delete | Task history belongs to the case audit trail |
| `case_notes` | Soft delete only (no updates) | Notes are immutable; soft delete with actor tracking is acceptable |
| `case_attachments` | Soft delete | Metadata remains; physical file cleanup is a separate process |

Soft delete columns on each of the above:

```
is_deleted   BOOLEAN     NOT NULL DEFAULT FALSE
deleted_at   TIMESTAMPTZ
deleted_by   UUID        REFERENCES users(id)
```

All queries against soft-deleted tables must include `WHERE is_deleted = FALSE` unless
intentionally auditing deleted records. The repository layer enforces this filter.

### Not applied to

| Table | Approach | Reason |
|---|---|---|
| `audit_logs` | Immutable — never deleted | Regulatory record |
| `case_status_history` | Immutable append-only | Operational ledger |
| `case_assignments` | Historical — closed with `unassigned_at` | Operational ledger |
| `case_notes` | Immutable — soft delete only, no updates | See Notes policy |
| Lookup tables | `is_active` flag | Reference data is deactivated, not deleted |
| `user_roles` | `removed_at` + `removed_by` (see below) | Role revocation requires an audit trail |
| `role_permissions` | Hard delete | Managed at system admin level; audit log covers it |

is possible but expensive and fragile. A dedicated column provides a direct, indexed
answer. Rows accumulate over time, representing the full role assignment history per
user.

**Ownership constraint:**

```sql
CONSTRAINT cases_subject_required
  CHECK (client_id IS NOT NULL OR organization_id IS NOT NULL)
```

A case must have at least one subject: a client, an organization, or both.
A case with neither is not a case — it is an unstructured task. That use case
belongs in a task tracker, not a case workflow system. The schema enforces this
at the database level, not just the service layer.

Valid combinations:
- `client_id` set, `organization_id` null → individual case, no org affiliation
- `client_id` null, `organization_id` set → org-level case, no specific individual
- Both set → case for a specific client within an organization

**Decision — `assigned_to_id` on `cases`:**
Denormalized for query efficiency. The most frequent query in the system is a worker's
open case list: `WHERE assigned_to_id = :userId AND is_deleted = FALSE`. This query
must not require a subquery or join through `case_assignments` history. The full
assignment history lives in `case_assignments`.

When a reassignment happens, the service layer must atomically:
1. Set `unassigned_at` and `unassigned_by` on the current active `case_assignments` row
2. Insert a new `case_assignments` row
3. Update `cases.assigned_to_id` and `cases.assigned_at`

All three steps must be in a single `@Transactional` method. Tests must validate this
invariant. See `case_assignments` for the database-level uniqueness guard.

**Decision — `case_number` format:**
Human-readable identifier in the format `CA-NNNNNN` (e.g., `CA-000001`). Generated by
the application service at case creation time using a PostgreSQL sequence:

```sql
CREATE SEQUENCE case_number_seq START 1;
```

```java
// In CaseService.create()
String caseNumber = "CA-" + String.format("%06d", nextval("case_number_seq"));
```

A global non-resetting sequence is used. The year is NOT embedded in the case number.
The original design used `CA-YYYY-NNNNNN` but this was rejected: a global sequence that
never resets but embeds a year creates an ambiguous number — operators expect
`CA-2026-000001` to mean the first case of 2026, not the N-th case in the system that
happened to be created in 2026. Using `CA-NNNNNN` is unambiguous. Creation year is
always available from `created_at`.

The sequence supports up to 999,999 cases before needing a format change. At target
scale of 500,000 cases this is adequate. Adjust padding to 7 digits proactively if
projections warrant it.

**Decision — `reopened_count`:**
Stored directly on the case. Cases reopened multiple times are a common operational
signal (overloaded workers, unclear initial resolution). Counting from
`case_status_history WHERE to_status.code = 'REOPENED'` is always possible but wastes
a join for a common reporting field.

#### `case_assignments`

Historical record of every assignment and reassignment.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| id | UUID | PK | |
| case_id | UUID | NOT NULL REFERENCES cases(id) | |
| assignee_id | UUID | NOT NULL REFERENCES users(id) | |
| assigned_by | UUID | NOT NULL REFERENCES users(id) | |

| unassigned_by | UUID | REFERENCES users(id) | |
| notes | TEXT | | Assignment or reassignment reason |

`unassigned_at IS NULL` identifies the currently active assignment. At most one row per
case should have `unassigned_at IS NULL`. This is enforced at the **database level**:

```sql
CREATE UNIQUE INDEX uq_case_assignments_one_active
  ON case_assignments (case_id)
  WHERE unassigned_at IS NULL;
```

Without this index, a bug in the service layer could create two active assignment
records for the same case with no database-level guard. The service layer closes the
prior record before inserting the new one, but the index is the authoritative constraint.

| created_by | UUID | NOT NULL REFERENCES users(id) | |

No `updated_at` or `updated_by`. Notes are not updated after creation.

**Notes policy:**
Notes represent observations at a point in time. In a compliance-oriented case
management system, editing notes is problematic: a legitimate typo correction is
indistinguishable from retroactive narrative manipulation. The schema removes the
option to edit.

**Correction workflow:**
If an operator needs to correct a note, they:
1. Soft-delete the incorrect note (`is_deleted = TRUE`, `deleted_by`, `deleted_at`)
2. Create a new note with the corrected content
3. Set `new_note.supersedes_note_id = old_note.id`

This preserves the full correction history. The UI can display superseded notes with a
"corrected" indicator. Audit log captures both the deletion and the new creation.

### Domain: Audit

#### `audit_logs`

Immutable record of all significant business actions. Append-only. Never updated or
deleted.

| Column | Type | Constraints | Notes |
|---|---|---|---|
| id | UUID | NOT NULL | Part of composite PK |
| occurred_at | TIMESTAMPTZ | NOT NULL DEFAULT now() | Part of composite PK |
| actor_id | UUID | REFERENCES users(id) | Nullable for system-initiated events |
| entity_type | VARCHAR(50) | NOT NULL | Domain constant: `case`, `user`, `case_task` |
| entity_id | UUID | NOT NULL | No FK — must reference deleted entities |
| action | VARCHAR(100) | NOT NULL | e.g. `case_status_changed`, `user_role_assigned` |
| old_values | JSONB | | State before the change |
| new_values | JSONB | | State after the change |
| metadata | JSONB | | IP address, session ID, user agent, etc. |

**Primary key:**

```sql
PRIMARY KEY (id, occurred_at)
```

`occurred_at` is included in the primary key to enable future range partitioning by
date. PostgreSQL declarative partitioning requires the partition key to be part of the
primary key. Adding `occurred_at` to the PK now avoids a costly migration later.

**Decision — JSONB for `old_values` / `new_values`:**
A strongly typed audit schema (one table per event type) is more directly queryable but
requires a schema migration for every new auditable event type. JSONB trades some query
ergonomics for unlimited extensibility. Audit trail queries are almost always filtered
by `entity_type + entity_id + occurred_at` range — not by value content — so JSONB
query complexity is not on the hot path.

**Decision — no FK on `entity_id`:**
The audit log must retain records for deleted entities. A FK would either prevent soft
deletion or cascade-delete audit history. Neither is acceptable.

**Decision — `entity_type` domain constants:**
`entity_type` uses stable, lowercase domain constant strings that map to table names —
not Java class names. Java classes get renamed during refactors; audit records do not.

| Domain constant | Table |
|---|---|
| `case` | cases |
| `user` | users |
| `organization` | organizations |
| `client` | clients |
| `case_task` | case_tasks |
| `case_note` | case_notes |
| `case_attachment` | case_attachments |
| `case_assignment` | case_assignments |
| `user_role` | user_roles |

**Standard `action` values:**

```
case_created               case_status_changed        case_assigned
case_reassigned            case_unassigned            case_closed
case_reopened              case_deleted               case_priority_changed
case_type_changed          case_due_date_changed
note_created               note_deleted
task_created               task_assigned              task_completed
task_deleted
attachment_uploaded        attachment_deleted
user_created               user_deactivated           user_deleted
user_role_assigned         user_role_removed
```

**Partitioning note:**
At 500K cases with an average of 10–15 audit events per case, the `audit_logs` table
will accumulate 5–7.5 million rows. At millions of rows, a single unpartitioned table
remains performant with the composite index on `(entity_type, entity_id, occurred_at)`.
If the system grows beyond projections, range partitioning by `occurred_at` (quarterly
or annually) can be added. The composite PK `(id, occurred_at)` ensures this partition
migration does not require a PK change.

**Cardinality:**
- A user may hold zero or more active roles
- A client may belong to zero or one organization
- A case must have at least one of client or organization (CHECK constraint)
- A case has exactly one current status, priority, and type
- A case has at most one current assignee (unique partial index enforces this)
- A case has zero or more: status transitions, assignment records, tasks, notes, attachments, audit events

## Case Lifecycle Model

```
                    ┌─────────────────────────────────────────┐
                    │                                         │
         ┌──────────▼──────────┐                             │
    ─────►        NEW          │                             │
         └──────────┬──────────┘                             │
                    │ assign                                  │
         ┌──────────▼──────────┐                             │
         │      ASSIGNED       │◄──────────── reassign ──────┤
         └──────────┬──────────┘                             │
                    │ begin review                            │
         ┌──────────▼──────────┐                             │
         │      IN_REVIEW      ├──── request info ──►┌───────┴──────┐
         └──────┬──────┬───────┘                     │ PENDING_INFO │
                │      │        ◄── info received ───└──────────────┘
                │      │
                │      └──── escalate ────►┌────────────┐
                │                          │  ESCALATED │
                │           ◄── resolved ──└────────────┘
                │
         ┌──────▼──────┐     ┌────────────┐
         │  APPROVED   │     │   DENIED   │
         └──────┬──────┘     └─────┬──────┘
                │                  │
         ┌──────▼──────────────────▼──────┐
         │            CLOSED              │
         └──────────────┬─────────────────┘
                        │ reopen
         ┌──────────────▼─────────────────┐
         │           REOPENED             ├──── re-enters ASSIGNED flow
         └────────────────────────────────┘
```

**Forbidden transitions (service layer enforces):**
- `NEW → CLOSED` (must go through assignment and review)
- `NEW → APPROVED` or `NEW → DENIED`
- `APPROVED ↔ DENIED` (terminal states cannot transition to each other)
- Any terminal state (`is_terminal = TRUE`) → any state other than `REOPENED`

**Implementation:**
The service layer queries `case_statuses.is_terminal` rather than hard-coding status
code strings. This means adding a new terminal status (e.g., `WITHDRAWN`) requires only
a seed migration row with `is_terminal = TRUE` — no service code change.

Transition rules themselves (which transitions are allowed) are encoded in a service
constant map, not in the database. This is appropriate: business rules belong in code,
not in a database table that becomes a hidden configuration system.

## Design Tradeoffs

### 1. UUID v7 via application layer

**Benefit:** Time-ordered inserts eliminate B-tree fragmentation. API exposure does not
leak sequential IDs. Consistent UUID type across all tables.

**Risk:** UUID generation moves from the database (`gen_random_uuid()`) to the
application. If application code fails to set the PK before insert, the database will
throw a NOT NULL violation — not a silent gap. This is acceptable.

**Dependency:** Requires `uuid-creator` library in the Spring Boot project. This is
a well-maintained, widely-used Java library. If a future PostgreSQL version adds native
UUID v7 generation, the change is isolated to the service/repository layer.

---

### 2. Notes are immutable

**Benefit:** Audit-clean. The timeline of what was known/said at each point in time
is preserved. `supersedes_note_id` allows correction workflows to be visible in the UI.

**Risk:** Operators cannot fix typos in-place. The correction UX (soft-delete + new note
+ supersedes link) is slightly more complex to implement than an edit endpoint.

**Not a problem for:** The schema. The service layer implements the correction workflow.
The schema simply removes the columns that would allow silent edits.

---

### 3. `assigned_to_id` denormalized on `cases`

**Benefit:** Worker dashboard queries are single-table with a composite index. No join
through assignment history required for the most frequent query in the system.

**Risk:** Consistency invariant requires `@Transactional` discipline. A bug that updates
`case_assignments` but not `cases.assigned_to_id` would produce incorrect dashboards
without immediate failure.

**Mitigation:** The `uq_case_assignments_one_active` unique partial index prevents
duplicate active assignment records. Service method tests should explicitly verify
that `cases.assigned_to_id` matches the active `case_assignments` record after each
reassignment.

---

### 4. Ownership CHECK constraint on `cases`

**Benefit:** The schema now defines what a valid case is. A case without a client or
organization is rejected at the database level, not discovered later in application
testing.

**Tradeoff:** Slightly reduces flexibility. Internal operational cases with no client
and no org affiliation cannot be created. If this use case emerges, the constraint can
be dropped or relaxed in a migration with documented justification.

---

### 5. `audit_logs` composite primary key `(id, occurred_at)`

**Benefit:** Enables future declarative range partitioning by `occurred_at` without
requiring a PK migration. The cost of carrying `occurred_at` in the PK is minimal.

**Risk:** The PK is slightly larger (UUID + TIMESTAMPTZ = 24 bytes vs 16 bytes for UUID
alone). Acceptable at the target scale.

---

### 6. `user_roles` revocation tracking

**Benefit:** Role revocation history is directly queryable without parsing JSONB from
`audit_logs`. "When was user X's SUPERVISOR role removed and who removed it?" is a
single indexed query.

**Cost:** Rows accumulate over time (one row per assignment/revocation pair per user).
At 1,000 users with occasional role changes, this table will never be large. Negligible.

---

### 7. No SLA, queue, or tag tables in Phase 1

The following were evaluated and deferred:

**SLA tracking:** Requires SLA rule definitions, breach tracking, and automatic
escalation triggers. This is Phase 6+ work. The `due_date` on cases is a sufficient
proxy for initial operational use.

**Work queues / queue ownership:** The unassigned pool (`assigned_to_id IS NULL`) is a
functional queue for initial use. Formal queue management with named queues, queue
ownership by team or supervisor, and queue assignment rules is a future enhancement.

**Tags/labels:** Many enterprise systems support case tags for flexible categorization.
`case_types` covers the primary categorization need. Tags can be added as a
`case_tags` lookup table + `case_tag_assignments` junction in a future phase.

## Migration Implementation Plan

**Phase 2 Flyway migration sequence:**

```
V1__create_trigger_function.sql        -- set_updated_at() trigger function
V2__create_users.sql                   -- users table + trigger
V3__create_roles_permissions.sql       -- roles, permissions tables
V4__create_user_roles_permissions.sql  -- user_roles, role_permissions + indexes
V5__create_lookup_tables.sql           -- case_statuses, case_priorities, case_types, task_statuses
V6__seed_lookup_data.sql               -- INSERT rows for all lookup tables
V7__create_sequence.sql                -- case_number_seq
V8__create_organizations_clients.sql   -- organizations, clients + triggers + indexes
V9__create_cases.sql                   -- cases table + CHECK constraint + trigger
V10__create_case_history.sql           -- case_status_history, case_assignments + indexes
V11__create_case_work.sql              -- case_tasks, case_notes, case_attachments + triggers + indexes
V12__create_audit_logs.sql             -- audit_logs + indexes
V13__create_remaining_indexes.sql      -- any composite indexes not created inline
```

**Migration rules:**
- Each migration is irreversible once applied to a non-development environment
- All migrations must execute cleanly from a fresh empty database
- `V6__seed_lookup_data.sql` must execute after `V5` and before any case-related tables
- Index migrations (`V13`) may be deferred if schema validation is the priority
- Never modify a committed migration — only add new ones
